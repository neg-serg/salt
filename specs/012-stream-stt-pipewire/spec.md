# Feature Specification: Streaming STT через PipeWire

**Feature Branch**: `012-stream-stt-pipewire`
**Created**: 2026-03-10
**Status**: Draft
**Input**: User description: "Нужно через whisper уметь распознавать аудиопоток непрерывно, например тот который идёт через PipeWire, посмотри как это реализовать"

## Clarifications

### Session 2026-03-10

- Q: Должна ли утилита поддерживать одновременный захват из двух источников (микрофон + выход приложения) для транскрипции обеих сторон разговора? → A: Да, два источника одновременно с пометкой кто говорит (source label) в выводе
- Q: Должна ли утилита показывать промежуточный текст по мере произнесения фразы, или только финальный результат после паузы? → A: Промежуточный текст обновляется в реальном времени, затем заменяется финальным

## User Scenarios & Testing

### User Story 1 — Непрерывное распознавание с микрофона (Priority: P1)

Пользователь запускает утилиту потокового распознавания речи, говорит в микрофон, и в реальном времени видит текст транскрипции в терминале. Распознавание работает непрерывно до остановки пользователем. Речь определяется автоматически (VAD), тишина не генерирует ложных транскрипций.

**Why this priority**: Базовый use case — микрофон самый частый источник аудио. Без этого остальные сценарии бессмысленны.

**Independent Test**: Запустить утилиту, произнести несколько фраз на русском и английском, убедиться что текст появляется в терминале с задержкой не более 3 секунд после конца фразы.

**Acceptance Scenarios**:

1. **Given** утилита запущена с настройками по умолчанию, **When** пользователь произносит длинную фразу, **Then** промежуточный текст начинает появляться в терминале во время произнесения, а финальный результат фиксируется в течение 3 секунд после паузы
2. **Given** утилита работает, **When** наступает тишина на 5+ секунд, **Then** никаких ложных транскрипций не генерируется
3. **Given** утилита работает, **When** пользователь произносит фразу на английском, **Then** текст распознаётся корректно (язык определяется автоматически)
4. **Given** утилита работает, **When** пользователь нажимает Ctrl+C, **Then** утилита корректно завершается

---

### User Story 2 — Распознавание аудио из приложений (Priority: P2)

Пользователь хочет распознавать аудиопоток из конкретного приложения (например, звонок в Telegram Desktop, аудио из браузера, подкаст). Утилита подключается к аудиовыходу выбранного приложения через PipeWire и транскрибирует в реальном времени. При транскрипции звонков утилита может захватывать два источника одновременно (микрофон + выход приложения), помечая каждый сегмент транскрипции меткой источника (например, «mic» / «app»).

**Why this priority**: Расширяет функциональность за пределы микрофона — позволяет транскрибировать любой аудиопоток в системе, включая обе стороны разговора.

**Independent Test**: Запустить воспроизведение аудио в браузере, указать утилите захватывать аудио из браузера, убедиться что транскрипция соответствует воспроизводимому содержимому.

**Acceptance Scenarios**:

1. **Given** играет аудио в приложении, **When** пользователь указывает утилите источник аудио (ID узла PipeWire или имя приложения), **Then** утилита захватывает этот аудиопоток и выдаёт транскрипцию
2. **Given** пользователь не знает ID устройства, **When** он запрашивает список доступных источников, **Then** утилита показывает список активных аудиопотоков с понятными именами
3. **Given** идёт звонок в приложении, **When** пользователь указывает два источника (микрофон + приложение), **Then** утилита захватывает оба потока и помечает каждый сегмент транскрипции меткой источника

---

### User Story 3 — Вывод транскрипции для интеграции (Priority: P3)

Пользователь хочет направить вывод транскрипции в другую программу: записать в файл, передать по pipe, или отправить как ввод в OpenClaw. Формат вывода должен быть структурированным и пригодным для автоматической обработки.

**Why this priority**: Без интеграционного вывода утилита полезна только для чтения глазами. С ним — становится строительным блоком для автоматизации.

**Independent Test**: Запустить утилиту с выводом в JSON, перенаправить в файл, проверить что каждая строка — валидный JSON с текстом и таймкодом.

**Acceptance Scenarios**:

1. **Given** утилита запущена с JSON-выводом, **When** распознаётся фраза, **Then** в stdout выводятся JSON-строки с полями: текст, временная метка, язык, источник, тип (partial/final)
2. **Given** утилита запущена с текстовым выводом, **When** распознаётся фраза, **Then** в stdout выводится только текст (одна фраза на строку)
3. **Given** вывод утилиты перенаправлен в pipe, **When** распознаётся фраза, **Then** данные появляются в pipe без буферизации

---

### Edge Cases

- Что происходит, если микрофон отключён или недоступен? → Понятное сообщение об ошибке с указанием как выбрать другой источник
- Что происходит при фоновом шуме (музыка, вентилятор)? → VAD должен фильтровать неречевые звуки; при высоком уровне шума — предупреждение
- Что происходит, если GPU занят другой задачей (недостаточно VRAM)? → Автоматическое переключение на CPU или понятное сообщение об ошибке
- Что происходит при очень длинной непрерывной речи (30+ минут)? → Утилита не должна накапливать память без ограничений; обработка по сегментам
- Что происходит, если целевое приложение (US2) завершается? → Утилита сообщает об отключении источника и завершается или ждёт переподключения

## Requirements

### Functional Requirements

- **FR-001**: Система MUST непрерывно захватывать аудиопоток из PipeWire-источника (микрофон по умолчанию)
- **FR-002**: Система MUST обнаруживать речевую активность (VAD) и транскрибировать только сегменты с речью
- **FR-003**: Система MUST выводить промежуточную транскрипцию в реальном времени по мере произнесения фразы, обновляя текст на месте; после паузы в речи промежуточный текст заменяется финальным результатом
- **FR-004**: Система MUST поддерживать автоопределение языка (как минимум EN и RU)
- **FR-005**: Система MUST позволять выбор источника аудио: микрофон по умолчанию, конкретное устройство по ID, или аудиопоток приложения. Система MUST поддерживать одновременный захват из двух источников (например, микрофон + выход приложения) с пометкой источника в каждом результате транскрипции
- **FR-006**: Система MUST предоставлять список доступных аудиоисточников PipeWire
- **FR-007**: Система MUST поддерживать два формата вывода: простой текст (по умолчанию) и JSON (по флагу)
- **FR-008**: Система MUST использовать локальную GPU-ускоренную модель распознавания (whisper.cpp с HIPBLAS, уже установлен)
- **FR-009**: Система MUST корректно завершаться по Ctrl+C с освобождением аудиоресурсов
- **FR-010**: Система MUST запускаться как CLI-утилита из терминала

### Key Entities

- **Audio Source**: Источник аудио в PipeWire — микрофон, виртуальный источник, или монитор выхода приложения. Характеризуется: ID узла, имя, тип (input/monitor), частота дискретизации
- **Speech Segment**: Непрерывный фрагмент речи, обнаруженный VAD. Характеризуется: начало, конец, аудиоданные
- **Transcription Result**: Результат распознавания сегмента. Характеризуется: текст, язык, временная метка, уверенность модели, метка источника (source label), тип (partial — промежуточный, final — окончательный)

## Success Criteria

### Measurable Outcomes

- **SC-001**: Транскрипция появляется в течение 3 секунд после окончания произнесённой фразы
- **SC-002**: Точность распознавания английской речи не ниже 95% (чистая речь, без шума)
- **SC-003**: Точность распознавания русской речи не ниже 90% (чистая речь, без шума)
- **SC-004**: Утилита работает непрерывно 30+ минут без деградации памяти или скорости
- **SC-005**: Ложные срабатывания на тишине: 0 за 5 минут молчания
- **SC-006**: Утилита запускается и готова к распознаванию менее чем за 10 секунд

## Assumptions

- PipeWire установлен и работает (стандарт для CachyOS/Arch Linux)
- whisper.cpp с HIPBLAS уже собран и whisper-cli доступен на PATH (feature 3)
- Модель ggml-large-v3-turbo (1.6GB) уже скачана в `speech-engines/voices/`
- GPU (RX 7900 XTX) имеет достаточно свободной VRAM (~4GB для whisper inference)
- Аудио формат для whisper: 16kHz, mono, 16-bit PCM

## Scope

### В скоупе

- CLI-утилита для потокового распознавания
- Захват аудио из микрофона и из приложений через PipeWire
- VAD для сегментации речи
- Текстовый и JSON вывод
- Использование существующей инфраструктуры whisper.cpp (feature 3)

### Вне скоупа

- GUI/TUI с визуализацией звуковой волны
- Интеграция с OpenClaw (будет отдельной задачей после стабилизации)
- Speaker diarization на основе голоса (определение кто говорит по голосовому отпечатку). Пометка источника (mic/app) — в скоупе
- Перевод речи на другой язык
- Работа как systemd-сервис (утилита запускается вручную)
