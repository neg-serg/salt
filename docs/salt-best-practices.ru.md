# Лучшие практики Salt и интеграция с LLM (2023-2026)

Справочный документ по лучшим практикам управления конфигурацией Salt с фокусом на masterless-установку на одной рабочей станции и разработку с помощью LLM. Каждая практика содержит статус аудита для данного проекта.

**Область**: Salt Project 3006-3008, masterless-режим, CachyOS (на базе Arch)
**Аудитория**: Разработчики, сопровождающие кодовую базу Salt + AI-инструменты (Claude Code, OpenCode)
**Связь с CLAUDE.md**: CLAUDE.md содержит *правила для AI* (предписывающие, краткие). Этот документ объясняет *почему эти правила существуют* и покрывает более широкие темы (здоровье экосистемы, альтернативы, стратегия тестирования). Ссылается на CLAUDE.md по разделам — не дублирует содержание.

**Маркеры статуса**: ✅ Уже следует | ⚠️ Частично следует | ❌ Ещё не внедрено | ➖ Неприменимо

## 1. Организация состояний

### SO-01: Файл-на-домен ✅

**Практика**: Каждый `.sls`-файл отвечает за один домен (audio, fonts, dns и т.д.). Новая функциональность добавляется в существующий доменный файл; новые `.sls`-файлы создаются только когда ни один существующий домен не подходит.

**Обоснование**: Механизм `include` в Salt создаёт неявную связность. Меньше хорошо ограниченных файлов — проще граф зависимостей и понятнее, что затронет `just apply desktop`.

**В проекте**: 44 файла состояний в `states/`, каждый покрывает отдельный домен. Оркестратор `states/system_description.sls` (строки 44-98) включает все домены в логическом порядке.

### SO-02: Макросы для повторяющихся паттернов ✅

**Практика**: Типовые инфраструктурные паттерны (установка пакетов, загрузка с GitHub, управление сервисами) ДОЛЖНЫ использовать макросы из `_macros_*.jinja`. Запрещено дублировать паттерн, уже реализованный макросом.

**Обоснование**: Макросы кодируют устойчивость к сети (retry), защиту идемпотентности (creates/unless), единообразное именование (install_*, build_*) и параллельное выполнение. Обход теряет все четыре гарантии.

**В проекте**: 5 файлов макросов покрывающих все повторяющиеся паттерны. См. раздел «Macros» в CLAUDE.md.

### SO-03: Соглашение об именовании State ID ✅

**Практика**: Использовать паттерн `target_descriptor` (например, `loki_config`, `greetd_enabled`). Никогда не использовать пути к файлам как State ID. Исключение: префиксы `install_*` и `build_*` зарезервированы для ID, сгенерированных макросами.

**Обоснование**: Описательные ID делают цепочки `require` читаемыми. Пути в качестве ID ломаются при перемещении файлов.

### SO-04: Пороги для встроенного содержимого ✅

**Практика**: Конфиги >= 10 строк — в `states/configs/`, юниты systemd — в `states/units/`, скрипты — в `states/scripts/`. Встроенное содержимое для коротких однострочников.

### SO-05: Централизованный импорт ✅

**Практика**: Единый файл `_imports.jinja` реэкспортирует все макросы и константы. Файлы состояний импортируют один файл вместо выборки из нескольких файлов макросов.

## 2. Паттерны данных

### DR-01: `import_yaml` вместо pillar в masterless ✅

**Практика**: В masterless-режиме использовать `import_yaml` для всех структурированных данных. Никогда не использовать pillar.

**Обоснование**: Ценность Pillar — шифрование при передаче и таргетирование per-minion через мастер. В masterless minion — единственный потребитель; накладные расходы pillar не дают никакой пользы. `import_yaml` рендерится локально, быстрее и проще для отладки.

**В проекте**: 20+ файлов данных в `states/data/`, загружаемых через `import_yaml`.

### DR-02: Конфигурация per-host через grains + host_config ✅

**Практика**: Использовать `grains['host']` для идентификации хоста. Поддерживать центральную карту конфигурации, объединяющую значения по умолчанию с per-host переопределениями.

**В проекте**: `states/host_config.jinja` загружает `data/hosts.yaml`, объединяет дефолты с per-host переопределениями, вычисляет производные поля.

### DR-03: Соглашения для файлов данных ✅

**Практика**: Списки пакетов в `states/data/*.yaml` ДОЛЖНЫ использовать формат YAML-списка (один пакет на строку) с инлайн-комментариями. Записи отсортированы лексикографически; логическая группировка допустима для больших наборов.

### DR-04: Граница Salt+Chezmoi с линт-проверкой ✅

**Практика**: Каждый конфиг-файл должен иметь ровно одного владельца — Salt или Chezmoi. Граница проверяется линтером `scripts/lint-ownership.py`.

## 3. Устойчивость к сети

### NR-01: Retry на всех сетевых операциях ✅

**Практика**: Каждое состояние, обращающееся к сети, ДОЛЖНО иметь `retry: {attempts: 3, interval: 10}`.

### NR-02: Параллельные независимые загрузки ✅

**Практика**: Использовать `parallel: True` на независимых состояниях загрузки/установки. Не использовать на состояниях с цепочками `require`.

### NR-03: Защита идемпотентности на каждой загрузке ✅

**Практика**: Каждое состояние загрузки/установки ДОЛЖНО иметь `creates:` или `unless:`.

### NR-04: Флаги curl: всегда `-fsSL` ✅

**Практика**: Все команды curl ДОЛЖНЫ использовать `-fsSL`: `-f` ошибка на HTTP-ошибках, `-sS` тихий режим с показом ошибок, `-L` следование редиректам.

## 4. Идемпотентность

### ID-01: Руководство по выбору guard ✅

**Практика**: Каждое состояние `cmd.run` и `cmd.script` ДОЛЖНО иметь guard:

| Guard | Когда использовать | Пример |
|-------|-------------------|--------|
| `creates:` | Состояние создаёт известный файл | `creates: /etc/udev/rules.d/50-qmk.rules` |
| `unless:` | Результат — изменение состояния системы | `unless: rg -qx 'steam' {{ pkg_list }}` |
| `onlyif:` | Состояние зависит от опционального ПО | `onlyif: command -v firewall-cmd` |

## 5. Тестирование и валидация

### TV-01: Валидация рендеринга перед apply ✅

**Практика**: Запускать `just validate` перед каждым `just apply`. Ошибка рендеринга в одном состоянии блокирует все остальные.

### TV-02: Пробный запуск с `test=True` ✅

**Практика**: Использовать `just test STATE` для пробного запуска, показывающего что БУДЕТ изменено.

### TV-03: Мульти-линтерный конвейер ✅

**Практика**: 8+ линтеров покрывающих YAML, Jinja, shell, Python, dotfiles, systemd, границы владения файлами.

### TV-04: Контейнерные smoke-тесты ✅

**Практика**: Тестирование рендеринга и выполнения состояний в изолированном контейнере (Podman + archlinux:latest).

### TV-05: Откат через снимки Btrfs ✅

**Практика**: Автоматическое создание снимков btrfs до и после каждого apply. Откат одной командой: `just rollback`.

## 6. Интеграция с LLM

### LM-01: CLAUDE.md как кодировщик конвенций ✅

**Практика**: Поддерживать полноценный CLAUDE.md в корне проекта со всеми конвенциями, ссылками на макросы, паттернами именования и требованиями к рабочему процессу.

**Обоснование**: AI-инструменты генерируют Salt-состояния на основе обучающих данных с общей документацией Salt. Проектные конвенции (имена макросов, паттерны guard, соглашения о путях) НЕ в обучающих данных. CLAUDE.md восполняет этот пробел.

### LM-02: Документация макросов для AI ✅

**Практика**: Документировать каждый макрос с именем, параметрами, назначением и примером использования.

### LM-03: Data-driven паттерны уменьшают ошибки AI ✅

**Практика**: При добавлении пакетов/инструментов/сервисов предпочитать добавление записей в YAML-файлы данных вместо написания новой логики состояний.

### LM-04: Цикл валидации ✅

**Практика**: После любого изменения, сгенерированного AI, запускать `just validate`.

### LM-05: Конституция как ограничители ✅

**Практика**: Поддерживать файл конституции с ненарушимыми принципами.

## 7. Анти-паттерны LLM

Типичные ошибки AI-инструментов при генерации Salt-состояний для этого проекта.

### AP-01: Использование pillar в masterless-режиме

**Причина**: Официальная документация Salt акцентирует pillar. LLM переоценивают значимость официальных документов.

**До** (неправильно):
```yaml
{% set pkgs = salt['pillar.get']('packages', []) %}
```

**После** (правильно):
```yaml
{% import_yaml 'data/packages.yaml' as packages %}
{{ pacman_install('base_packages', packages.base | join(' ')) }}
```

### AP-02: Отсутствие guard идемпотентности на `cmd.run`

**До** (неправильно):
```yaml
install_tool:
  cmd.run:
    - name: curl -fsSL https://example.com/tool -o /usr/local/bin/tool
```

**После** (правильно):
```yaml
install_tool:
  cmd.run:
    - name: curl -fsSL https://example.com/tool -o /usr/local/bin/tool
    - creates: /usr/local/bin/tool
    - retry:
        attempts: {{ retry_attempts }}
        interval: {{ retry_interval }}
```

### AP-03: `file.managed` без `replace: False` для самомодифицирующихся конфигов

**Причина**: LLM не знают, какие инструменты перезаписывают свои конфиги при запуске. Использовать `replace: False` для таких файлов.

### AP-04: Захардкоженные пути вместо переменных host_config

**До**: `- name: /home/alice/.config/myapp/config.toml`
**После**: `- name: {{ home }}/.config/myapp/config.toml`

### AP-05: Создание новых `.sls` вместо расширения существующих доменов

**Причина**: LLM трактуют каждый запрос как изолированный. Добавлять в существующие доменные файлы.

### AP-06: Использование pip/npm/cargo вместо макросов pacman

**До**: `cargo install bat`
**После**: `{{ pacman_install('bat', 'bat') }}`

### Оценка рисков для данного проекта

**Уровень риска: НИЗКИЙ**

Проект хорошо изолирован от проблем экосистемы Salt:
1. **Только базовые модули**: `file`, `cmd`, `service`, `pkg` — остаются в ядре Salt
2. **Masterless-режим**: Нет зависимости от инфраструктуры Salt-мастера
3. **Одна рабочая станция**: Нет управления флотом, таргетирования minion-ов
4. **Самодостаточное дерево состояний**: Под контролем версий, без внешних формул

## 9. Оценка альтернатив

| Инструмент | Сильные стороны | Слабые стороны | Осуществимость миграции |
|------------|----------------|----------------|------------------------|
| **Ansible** | Безагентный (SSH), большое сообщество, MCP-серверы | Медленнее, только YAML-логика | **Средняя** |
| **pyinfra** | Python-нативный, быстрый, современный | Маленькое сообщество | **Средняя** |
| **NixOS** | Истинная декларативность, атомарные откаты | Уже мигрировали с Nix | **Низкая** |
| **Pulumi** | AI-нативный IaC | Облачный фокус, избыточен для рабочей станции | **Низкая** |

**Рекомендация**: Оставаться на Salt. При необходимости миграции — pyinfra как наиболее жизнеспособная альтернатива.
