# Исследование миграции на pyinfra: Salt vs pyinfra для одного хоста

**Дата**: 2026-03-15
**Рекомендация**: **НЕ МИГРИРОВАТЬ** — pyinfra не дает значимого ускорения для masterless-режима на одном хосте и вносит критические регрессии.

## Резюме

| Фактор | Значение |
|--------|----------|
| Ускорение (полный apply) | **~0%** — узкое место в внешних командах (pacman, curl, makepkg) |
| Ускорение (идемпотентный apply) | **~5-6с** быстрее (12с → 6-7с) — только overhead запуска |
| Регрессия (потеря параллелизма) | **+30 мин** на свежем apply (скачивание моделей ollama + video_ai) |
| Трудозатраты на миграцию | **3-5 недель** (35 макросов, 41 файл состояний, 173 вызова макросов) |
| Риск регрессий | **Средне-высокий** — паттерны watch/onchanges, устаревание фактов |
| Bus factor pyinfra | **1** разработчик (Nick Barrett) vs 500+ контрибьюторов Salt |
| Итоговая выгода | **Отрицательная** — затраты значительно превышают пользу |

## 1. Сравнение скорости

### 1.1 Методология

Измерения на основе системы профилирования Salt (`scripts/state-profiler.py --trend`) по 126 логам apply на рабочей станции CachyOS. Оценки pyinfra теоретические (оба инструмента выполняют идентичные внешние команды через subprocess).

### 1.2 Идемпотентный (без изменений) apply

| Фаза | Salt | pyinfra (оценка) | Разница |
|------|------|-------------------|---------|
| Запуск (загрузка модулей, grains) | 5-6с | 1-2с | -4с |
| Компиляция состояний (Jinja+YAML) | 2-3с | 0с (импорт Python) | -2с |
| Выполнение (566 проверок) | 4.3с | ~4-5с | ~0с |
| **Итого** | **~12с** | **~6-7с** | **~5-6с быстрее** |

### 1.3 Регрессия параллельного выполнения

Salt поддерживает `parallel: True` для независимых состояний на одном хосте. pyinfra — **нет** (параллелизм только между хостами).

| Параллельное состояние | Файл | Последовательно | Параллельно | Регрессия |
|------------------------|------|-----------------|-------------|-----------|
| `pull_<model>` (5+ моделей) | ollama.sls | ~15 мин | ~5 мин | **+10 мин** |
| `video_ai_download_<model>` | video_ai.sls | ~30 мин | ~10 мин | **+20 мин** |
| **Итого** | | | | **+30 мин** |

### 1.4 Вердикт по скорости

| Сценарий | Salt | pyinfra | Победитель |
|----------|------|---------|------------|
| Идемпотентный apply | 12с | ~6-7с | pyinfra (+5с) |
| Свежий apply (без параллельных) | 5-10 мин | 5-10 мин | Ничья |
| Свежий apply (с моделями) | 15-20 мин | 45-50 мин | **Salt (+30 мин)** |

## 2. Матрица функциональных пробелов

| Функция Salt | pyinfra эквивалент | Критичность |
|---|---|---|
| `parallel: True` | **Нет** (один хост) | **Критическая** |
| `watch:`/`onchanges:` | `OperationMeta.did_change` | **Высокая** — многословно |
| `require`/`require_in` | Ручной порядок Python | Средняя |
| `creates:`/`unless:` | `_if` callable | Средняя |
| `import_yaml` | `yaml.safe_load()` | Тривиальная |
| Jinja макросы (35 шт.) | Python функции | Средняя (трудозатраты) |
| 400+ модулей состояний | ~44 операции | Средняя |

### Проблема двухфазной модели pyinfra

pyinfra собирает факты на этапе "подготовки" и **не обновляет их между операциями**. Это ломает типичные паттерны установка→настройка→запуск.

## 5. Рекомендация

### Решение: НЕ МИГРИРОВАТЬ

Все пять критериев оценки провалены:

| Критерий | Порог | Результат | Пройден? |
|----------|-------|-----------|----------|
| Ускорение (свежий apply) | >20% | ~0% | **НЕТ** |
| Регрессия скорости | <5% | +100-200% | **НЕТ** |
| Трудозатраты | <2 недель | 3-5 недель | **НЕТ** |
| Риск экосистемы | Сравнимый или лучше | Хуже (bus factor 1) | **НЕТ** |
| Паритет функций | Без критических пробелов | 1 критический, 2 высоких | **НЕТ** |

### Рекомендуемая альтернатива: оптимизация существующего Salt

1. Аудит для дополнительных `parallel: True` — много независимых состояний установки могут работать параллельно
2. Профилирование через `just profile-trend` — выявление реально медленных состояний
3. Предварительная сборка пакетов — кеширование PKGBUILD, которые не менялись
4. Улучшение идемпотентности — замена медленных `unless:` shell-проверок на файловые маркеры версий
