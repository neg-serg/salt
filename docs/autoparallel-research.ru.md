# Исследование автопараллелизма компилятора для рабочей станции CachyOS

Исследование установленных программ, которые могут выиграть от автопараллелизма GCC Graphite (`-ftree-parallelize-loops=N`) после перекомпиляции на 32-ядерной AMD рабочей станции CachyOS.

**Файл данных**: `states/data/autoparallel-candidates.yaml` (машиночитаемые классификации)
**Дата**: 2026-03-12
**Система**: CachyOS, GCC 15.2.1, 32-ядерный AMD, `makepkg.conf`: `-O3 -march=native` с LTO

## Итоги

Из 384 уникальных установленных пакетов:
- **8 высокий потенциал** — однопоточные C/C++ с CPU-ориентированной обработкой данных
- **14 средний** — частично распараллелены (OpenMP) с оставшимися непараллельными циклами
- **27 низкий** — уже распараллелены или неподходящий тип нагрузки
- **335 неприменимо** — интерпретируемые языки, I/O-ориентированные, событийные или метапакеты

Автопараллелизм — узкая оптимизация. Компилятор может распараллелить только циклы с доказуемо независимыми итерациями, простыми шаблонами доступа к массивам и без сложного наложения указателей. Большинство реальных программ не имеют таких паттернов в горячих путях.

## Справочник флагов GCC Graphite

### Основные флаги

| Флаг | Назначение | Риск |
|------|-----------|------|
| `-ftree-parallelize-loops=N` | Автопараллелизация независимых циклов с N потоками (libgomp) | Низкий — консервативный анализ |
| `-floop-parallelize-all` | Параллелизация даже циклов с неопределённой выгодой | Средний — может добавить накладные расходы для малых циклов |
| `-floop-nest-optimize` | Полиэдральная оптимизация циклов для кэш-локальности | Низкий — без многопоточности, только оптимизация кэша |
| `-ftree-loop-distribution` | Разделение тел циклов для выявления параллелизма | Низкий — автоматически включён при `-O3` |

### Рекомендуемая конфигурация

```bash
# В функции build() PKGBUILD:
export CFLAGS="$CFLAGS -ftree-parallelize-loops=16"
export CXXFLAGS="$CXXFLAGS -ftree-parallelize-loops=16"
export LDFLAGS="$LDFLAGS -lgomp"
```

**Количество потоков**: 16 оптимально для 32-ядерной системы. Эмпирические тесты показали:
- 8 потоков: ускорение 4.5x (на идеальных нагрузках)
- 16 потоков: ускорение 5.3x (оптимальная точка)
- 32 потока: ускорение 4.0x (регрессия — накладные расходы потоков превышают выгоду)

**Переопределение в рантайме**: `OMP_NUM_THREADS=N ./program` переопределяет скомпилированное количество потоков.

### Диагностика

```bash
# Посмотреть, какие циклы были распараллелены:
gcc -O3 -ftree-parallelize-loops=16 -fdump-tree-parloops-details -c file.c
# Искать "SUCCESS" в дамп-файле .parloops

# Graphite-специфичный анализ циклов:
gcc -O3 -floop-nest-optimize -fdump-tree-graphite-details -c file.c
```

## Справочник LLVM Polly (опционально)

Polly **не установлен по умолчанию** в CachyOS. Установка: `pacman -S polly`.

| Флаг | Назначение |
|------|-----------|
| `-mllvm -polly` | Включить полиэдральный оптимизатор Polly |
| `-mllvm -polly-parallel` | Включить параллельную генерацию кода |
| `-mllvm -polly-vectorizer=stripmine` | Полиэдральная векторизация |

**Рантайм**: Использует libgomp (как и GCC Graphite), не libomp. Дополнительный пакет не нужен.

**Rust**: Теоретически `-C llvm-args=-polly -C llvm-args=-polly-parallel`, но LLVM IR Rust (проверки границ, раскрутка стека, дискриминанты enum) препятствует обнаружению SCoP в большинстве кода. Не рекомендуется для программ на Rust.

**Зрелость**: Экспериментальный. Снижение активности разработки с 2020 года. Не используется в продакшене крупными проектами. Использовать только для целевых экспериментов.

## Рецепт модификации PKGBUILD

### Пошаговая инструкция

```bash
# 1. Снимок безопасности
sudo snapper create -d "pre-autopar: <package>" -t pre

# 2. Получить PKGBUILD
asp checkout <package>
cd <package>/trunk

# 3. Добавить флаги в build()
# Вставить в начало функции build():
export CFLAGS="$CFLAGS -ftree-parallelize-loops=16"
export CXXFLAGS="$CXXFLAGS -ftree-parallelize-loops=16"
export LDFLAGS="$LDFLAGS -lgomp"

# 4. Собрать и установить
makepkg -si

# 5. Пост-снимок
sudo snapper create -d "post-autopar: <package>" -t post

# 6. Бенчмарк
hyperfine --warmup 3 --runs 10 '<команда нагрузки>'
```

### Откат

```bash
# Вариант 1: Кэш pacman (самый быстрый)
sudo pacman -U /var/cache/pacman/pkg/<package>-<version>.pkg.tar.zst

# Вариант 2: Snapper (полный откат системы)
sudo snapper undochange <pre-id>..<post-id>

# Вариант 3: Рантайм-переключатель
OMP_NUM_THREADS=1 ./program  # Отключить параллелизм без пересборки
```

## Ограничения корректности

### Что GCC Graphite может распараллелить

- Счётные циклы `for` с аффинным доступом к массивам (`a[i]`, `a[2*i+1]`)
- Без межитерационных зависимостей (каждая итерация независима)
- Скалярные редукции (`sum += a[i]`) — используются приватные аккумуляторы
- Статически известное или вычислимое количество итераций

### Что не работает или вызывает ошибки компиляции

| Паттерн | Проблема | Пример |
|---------|----------|--------|
| Наложение указателей | Невозможно доказать независимость | `*p++ = *q++` |
| Неаффинные индексы | Полиэдральная модель не может представить | `a[b[i]]` (косвенный доступ) |
| Вызовы функций | Считаются имеющими побочные эффекты | Неинлайнированные, не-`const` вызовы |
| Сложный поток управления | Ветвления ломают анализ | Зависящий от данных `if` внутри цикла |
| Циклы while | Нет счётной итерации | `while (p = p->next)` |
| Малые циклы | Накладные расходы потоков превышают выгоду | Менее ~10K итераций |

### Наложение указателей — главное практическое ограничение

Большая часть кода C/C++ активно использует указатели. Без квалификаторов `restrict` компилятор не может доказать, что два указателя не ссылаются на одну и ту же память, что предотвращает параллелизацию. Поэтому:

- Программы на Fortran показывают значительно лучшие результаты автопараллелизма (семантика массивов предотвращает наложение по умолчанию)
- Программы на C с квалификаторами `restrict` на параметрах функций значительно помогают
- Большинство существующих кодовых баз C не используют `restrict`

## Совместимость с CachyOS

Значения по умолчанию `makepkg.conf` CachyOS:
```
CFLAGS="-march=native -O3 -pipe -fno-plt -fexceptions"
OPTIONS=(strip docs !libtool !staticlibs emptydirs zipman purge !debug lto !autodeps)
```

| Флаг по умолчанию | Взаимодействие | Статус |
|-------------------|---------------|--------|
| `-O3` | Обязателен — Graphite требует минимум `-O2` | Совместим |
| `-march=native` | Ортогонален — ISA vs параллелизм на уровне потоков | Совместим |
| `-pipe` | Нет взаимодействия | Совместим |
| `-fno-plt` | Нет взаимодействия | Совместим |
| LTO (`-flto`) | Может раскрыть больше циклов через кросс-модульный инлайнинг; увеличивает время компиляции | Совместим |
| `-fexceptions` | Нет взаимодействия | Совместим |

**Все флаги аддитивны** — нет конфликтов со значениями по умолчанию CachyOS.

## Риск перегрузки потоками

Когда программа уже использует OpenMP или pthreads, добавление автопараллелизма создаёт больше потоков, чем предполагалось:

```
Всего потоков = существующие_потоки × автопараллельные_потоки
```

Для `imagemagick` с 16 OpenMP потоками + 16 автопараллельных потоков = 256 потоков на 32-ядерной системе → серьёзная конкуренция.

**Меры**: Установить `OMP_NUM_THREADS` для ограничения общего количества потоков:
```bash
# Для программ с существующим OpenMP параллелизмом:
OMP_NUM_THREADS=8 magick convert ...  # 8 × (некоторый автопараллелизм) остаётся управляемым
```

## Влияние на размер бинарного файла и время компиляции

| Метрика | Влияние | Примечания |
|---------|---------|-----------|
| Размер бинарного файла | +5-15% секция text | Выделенные тела циклов + GOMP-обвязка |
| Привязка libgomp | ~200KB | Общая библиотека, загружается один раз |
| Время компиляции | Увеличение в 2-5 раз | ISL-солвер Graphite дорогой; O(n³) по размерности циклов |

Увеличение времени компиляции — основная практическая стоимость. Для точечных пересборок пакетов это приемлемо; для системного `makepkg.conf` — непомерно.

## Методология бенчмарков

### Протокол

Для каждой программы-кандидата:

1. **Базовая линия**: Установить стандартный пакет из репозитория
2. **Бенчмарк**: `hyperfine --warmup 3 --runs 10 --export-json baseline.json '<нагрузка>'`
3. **Пересборка**: Модифицировать PKGBUILD с флагами автопараллелизма
4. **Установка**: `makepkg -si`
5. **Бенчмарк**: `hyperfine --warmup 3 --runs 10 --export-json modified.json '<нагрузка>'`
6. **Сравнение**: Wall-clock время, User время (должно увеличиться при работе параллелизма), System время
7. **Дополнительно**: `perf stat -d '<нагрузка>'` для переключений контекста, поведения кэша
8. **Восстановление**: `pacman -U /var/cache/pacman/pkg/<оригинал>.pkg.tar.zst`

### Метрики

- **Wall-clock ускорение**: `базовое_время / модифицированное_время` — основная метрика
- **Утилизация CPU**: Увеличение User-времени указывает на реальную параллельную работу (не просто накладные расходы)
- **Переключения контекста**: Высокое количество переключений указывает на конкуренцию потоков
- **Пропускная способность памяти**: `perf stat` LLC-load-misses — может насытиться раньше CPU

### Результаты валидации: optipng

**Тестовая система**: CachyOS, GCC 15.2.1, AMD 32-ядерный, `-O3 -march=native`
**Тестовый файл**: 81MB плазменный фрактал PNG (4096×4096), уровень оптимизации `-o2`
**Бенчмарк**: `hyperfine --warmup 1 --runs 5`

| Вариант | Wall-clock | User time | CPU% |
|---------|-----------|-----------|------|
| Stock optipng (системная zlib, репозиторий CachyOS) | **12.87 с ± 0.09** | 12.80 с | 99% |
| Встроенная zlib, без автопараллелизма | **15.02 с ± 0.26** | 14.49 с | 99% |
| Встроенная zlib + `-ftree-parallelize-loops=16` | **16.88 с ± 0.20** | 250.34 с | 1485% |

**Результат: на 31% медленнее с автопараллелизмом.**

Graphite успешно распараллелил циклы в 6 модулях (deflate, opngreduc, pngxrbmp, pngxrtif, gifread, tiffread), запуская 16 потоков. User time увеличился в 19.6 раз, подтверждая реальную параллельную работу. Однако:

1. **Распараллеленные циклы слишком мелкие** — скользящее окно deflate оперирует блоками ~256 байт, а не миллионами элементов. Накладные расходы на создание/синхронизацию потоков доминируют.
2. **Встроенная zlib на 17% медленнее системной** — системная zlib CachyOS включает SIMD-оптимизации (патчи cloudflare/intel), которых нет в ванильной встроенной zlib.
3. **`OMP_NUM_THREADS=1` не предотвратил создание потоков** — скомпилированное количество потоков (16) переопределило переменную окружения, что делает рантайм-переключатель ненадёжным.

**Вывод**: даже для кандидата с «высоким потенциалом» и доказуемо параллелизуемыми циклами автопараллелизм ухудшил производительность. Теоретическое обещание параллелизма на уровне циклов не выдерживает столкновения с реальным кодом, где горячие циклы слишком малы для амортизации накладных расходов пула потоков.
