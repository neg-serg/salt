## Результаты анализа Phase 1 (T1.1–T1.9)

### T1.1 Идемпотентность — ПРОЙДЕНО

Все 59 состояний cmd.run/cmd.script имеют корректные guards:
- `creates:` — 18 состояний (проверка файлового артефакта)
- `unless:` — 28 состояний (проверка условия)
- `onchanges:` — 8 состояний (событийный запуск)
- `onlyif:` — 3 состояния (условное выполнение)
- `stateful: True` — 1 состояние (pacman_db_warmup, самоотчёт)
- `prereq:` — 1 состояние (остановка transmission перед изменением конфига)

Все 3 использования `replace: False` обоснованы (самомодифицирующиеся конфиги: kanata, AdGuardHome, OpenClaw).
Сервисные паттерны корректны: сокет-активируемые сервисы (libvirtd, pcscd) используют только `service.enabled`.
Циклов перезапуска не обнаружено.

### T1.2 Зависимости — ПРОЙДЕНО

25+ межфайловых ссылок require/watch/onchanges проверены — все указывают на существующие state ID.
ID, генерируемые макросами (install_*, *_enabled, *_daemon_reload), сверены с шаблонами.
Циклических зависимостей нет. Неявных зависимостей от порядка нет.
Все цепочки pkg.installed → service.enabled корректны.
Все связки file.managed → service.running с watch на месте.
Условные зависимости корректно обрабатываются через Jinja feature flags.

### T1.3 Сетевая устойчивость — ПРОЙДЕНО

Все сетевые операции используют макросы со встроенным retry (`attempts: 3, interval: 10`),
флагами `curl -fsSL` и `parallel: True` где безопасно. Таймауты адекватны:
- Стандартные загрузки: 60с
- Загрузка моделей Ollama: 660с (большие модели)
- Сборка контейнера Amnezia: 3600с
- Сборка Tidal SuperCollider: 1200с

Нет `parallel: True` на состояниях с require-цепочками к другим сетевым состояниям.

### T1.4 Health Checks — ПРОЙДЕНО

Все сервисы из критериев приёмки G4 имеют health checks:
- **dns.sls:** Unbound (`unbound-control status`, 30с), AdGuardHome (`GET /`, 30с)
- **monitoring.sls:** Loki (`GET /ready`, 30с), Promtail (`GET /ready`, 30с), Grafana (`GET /api/health`, 30с)
- **ollama.sls:** Ollama (`GET /api/tags`, 30с)
- **llama_embed.sls:** Llama-embed (`GET /health`, 90с — увеличен для загрузки GPU-модели)
- **services.sls:** Jellyfin (`GET /health`, 30с), Transmission (`GET /transmission/web/`, 30с)

Сервисы без health checks (обоснованно):
- Avahi: пассивное обнаружение сервисов, без проблем запуска
- Bitcoind: ручной запуск (`enabled=False`)
- DuckDNS: запуск по таймеру, нет постоянного сервера
- Samba: ручной запуск (`service_stopped`)
- Xray/sing-box: нет встроенного endpoint для проверки здоровья

### T1.5 Безопасность — ПРОЙДЕНО

Plaintext-секретов в state-файлах не обнаружено — все используют ссылки на gopass.
Права на файлы корректны: openclaw.json (0600), mpdasrc (0600), sudoers (0440).
Токен DuckDNS передаётся через stdin (`curl -d @-`), не через аргументы командной строки.
Разрешения sudoers ограничены конкретными бинарниками.
Udev-правила используют безопасные паттерны (MODE, GROUP, TAG+="uaccess").

### T1.6 Стиль — ПРОЙДЕНО (98.3%)

115+ state ID проверены — 98.3% следуют паттерну `target_descriptor`.
2 незначительных отклонения: `qmk_udev_rules_reload` (суффикс действия), `transmission_stop_before_settings_change` (избыточная длина).
Нет inline-реализаций, дублирующих функциональность макросов.
Все пути используют `{{ home }}`/`{{ user }}` — ноль захардкоженных `/home/neg`.
Все блоки inline-контента адекватного размера (<30 строк).
Все 17 ссылок `import_yaml` указывают на существующие файлы.

### T1.7 Systemd-юниты — ПРОЙДЕНО

Проанализирован 21 юнит (10 системных, 11 пользовательских).
Все enabled-сервисы имеют секции `[Install]` (duckdns-update.service намеренно без неё — запускается таймером).
Политики перезапуска настроены: `on-failure` (большинство), `always` (ollama, llama-embed).
Все сетевые сервисы имеют `After=network-online.target`.
Hardening применён: NoNewPrivileges (20/21), RestrictNamespaces (19/21), PrivateTmp (19/21).
ProtectHome отсутствует у ollama/llama-embed (нужен доступ к моделям) — принятый риск.
PrivateDevices отсутствует у fancontrol/sing-box (нужен доступ к устройствам) — принятый риск.
Конфигурации таймеров корректны: `Persistent=true` на всех таймерах.

### T1.8 Макросы — ПРОЙДЕНО

34 макроса в 5 файлах. Все имеют:
- Корректные guards `creates:`/`unless:`, соответствующие реальным артефактам
- `retry: {attempts: 3, interval: 10}` на сетевых операциях
- `parallel: True` где уместно
- Корректное экранирование shell — векторов инъекции не обнаружено

`gopass_secret` использует `python_shell=True` с конкатенацией строк, но пути ключей —
всегда статические литералы (напр., `'api/anthropic'`) — принятый риск с документацией.
Неиспользуемых макросов не обнаружено.
Конфликтов state ID при раскрытии макросов нет — все ID пространственно разделены параметром name.

### T1.9 Data-файлы — ПРОЙДЕНО

17 YAML data-файлов проверены — все синтаксически корректны и активно потребляются.
Осиротевших записей нет. Дубликатов в файлах нет.
`feature_matrix.yaml` изначально помечен как неиспользуемый — фактически потребляется
`host_config.jinja` для тестирования render-matrix (синтетические сценарии хостов для CI).
`hosts.yaml` — источник продуктивной конфигурации хостов, не устаревший.
25 версий зафиксированы в `versions.yaml`, все используются через подстановку `${VER}`.

## Принятые риски

**F06 — Ollama/Llama-embed без ProtectHome:**
Эти сервисы загружают ML-модели из `$HOME/.ollama` и `/mnt/one`. Добавление `ProtectHome=read-only`
потребовало бы перемещения хранилища моделей в системную директорию. Текущая конфигурация
соответствует upstream-значениям Ollama.

**F07 — Fancontrol/Sing-box без PrivateDevices:**
Fancontrol читает аппаратные сенсоры через `/sys/class/hwmon`. Sing-box создаёт TUN-сетевые адаптеры.
Оба требуют прямого доступа к устройствам по дизайну.

**F09 — gopass_secret python_shell:**
Макрос конкатенирует `'gopass show -o ' ~ key`, где `key` — всегда захардкоженная строка
типа `'api/anthropic'` или `'lastfm/password'`. Пользовательский ввод не достигает этого пути.
Задокументировано в CLAUDE.md memory как ограничение конвенции.
