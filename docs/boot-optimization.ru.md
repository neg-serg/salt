# Руководство по оптимизации загрузки

Практическое руководство по сокращению времени загрузки рабочей станции CachyOS, управляемой Salt.

---

## 1. Обзор времени загрузки

Базовые измерения выполнены с помощью `systemd-analyze` и `systemd-analyze blame` при типичной холодной загрузке.

| Фаза | Длительность | Примечания |
|---|---|---|
| Прошивка (UEFI POST) | 20.0с | Основной вклад; зависит от оборудования |
| Загрузчик (Limine) | 1.3с | Таймаут уже установлен в 1с |
| Ядро | 2.6с | Включая декомпрессию и раннюю инициализацию |
| initrd | 3.1с | mkinitcpio со сжатием zstd |
| Пространство пользователя | 3.4с | Службы systemd до graphical.target |
| **Итого** | **30.5с** | Холодная загрузка, прошивка доминирует |

Прошивка составляет ~65% общего времени загрузки. Разделы ниже нацелены на каждую фазу.

## 2. Настройки UEFI (ROG CROSSHAIR X870E EXTREME)

Эти изменения вносятся в настройках BIOS/UEFI материнской платы (Del при POST). Настройки указаны с точными путями меню для BIOS 1715+ (AGESA ComboAM5 PI 1.2.0.3g+).

### Группа A: Память (экономия 5-15с)

Обучение памяти — основной фактор времени POST на AM5. DDR5 на 6000 МТ/с с 2x32 ГБ DIMM требует обширной калибровки UMC при каждой холодной загрузке. Memory Context Restore (MCR) кеширует подтверждённые результаты обучения в энергонезависимой памяти и восстанавливает их при последующих загрузках.

| # | Настройка | Путь в UEFI | Значение | Выигрыш |
|---|---|---|---|---|
| 1 | Memory Context Restore | `Advanced > AMD CBS > UMC Common Options > DDR Options > DDR Memory Features > Memory Context Restore` | **Enabled** | 5-15с |
| 2 | Power Down Enable | `Advanced > AMD CBS > UMC Common Options > DDR Options > DDR Controller Configuration > DDR Power Options > Power Down Enable` | **Enabled** | (спутник MCR) |
| 3 | DDR Training Runtime Reduction | `Advanced > AMD CBS > UMC Common Options > DDR Options > DDR Training Options > DDR Training Runtime Reduction` | **Enabled** | 0-2с |

**Важно**: MCR ДОЛЖЕН быть включён совместно с Power Down Enable — документация AMD подтверждает нестабильность (BSOD, повреждение памяти) без него. Отключайте MCR временно при замене планок RAM или изменении таймингов памяти для принудительного переобучения. Сброс CMOS восстанавливает работу при любых проблемах, связанных с MCR.

### Группа B: Неиспользуемое оборудование (экономия 2-4с)

Каждая отключённая подсистема убирает время инициализации из POST. Эти настройки отключают оборудование, которое не используется.

| # | Настройка | Путь в UEFI | Значение | Выигрыш |
|---|---|---|---|---|
| 4 | SATA Controller(s) | `Advanced > SATA Configuration > SATA Controller(s)` | **Disabled** | 0.5-1с |
| 5 | Network Stack | `Advanced > Network Stack Configuration > Network Stack` | **Disabled** | 1-3с |
| 6 | Legacy USB Support | `Advanced > USB Configuration > Legacy USB Support` | **Auto** | 1-2с |

**Примечания**: SATA можно отключить, т.к. все накопители — NVMe (проверено через `lsblk`). Network Stack управляет только загрузкой по PXE — сетевые карты остаются функциональными в ОС. Legacy USB установлен в Auto (не Disabled), чтобы сохранить доступ клавиатуры в BIOS.

### Группа C: Конфигурация загрузки (экономия 1-3с)

Быстрые изменения с минимальным риском, устраняющие задержки, сканирование устаревших устройств и ненужные визуальные элементы загрузки.

| # | Настройка | Путь в UEFI | Значение | Выигрыш |
|---|---|---|---|---|
| 7 | Fast Boot | `Boot > Boot Configuration > Fast Boot` | **Enabled** | 2-5с |
| 8 | Post Delay Time | `Boot > Boot Configuration > Post Delay Time` | **0 sec** | 0-5с |
| 9 | Boot Logo Display | `Boot > Boot Configuration > Boot Logo Display` | **Disabled** | 0-1с |
| 10 | Wait For F1 If Error | `Boot > Boot Configuration > Wait For F1 If Error` | **Disabled** | 0с (предотвращает зависание) |
| 11 | Launch CSM | `Boot > CSM > Launch CSM` | **Disabled** | 0-2с |
| 12 | Core Watchdog Timer | `Advanced > AMD CBS > CPU Common Options > Core Watchdog > Core Watchdog Timer Enable` | **Disabled** | незначительно |

### Откат

Перед внесением изменений сохраните профиль BIOS через `Tool > ASUS User Profile > Save to Profile`.

- **Клавиатура работает**: войдите в BIOS → `Tool > ASUS User Profile > Load from Profile` → восстановите сохранённый слот
- **Клавиатура не работает в BIOS**: нажмите кнопку CMOS Clear на задней панели (сбрасывает все настройки по умолчанию)
- **Не проходит POST**: используйте кнопку USB BIOS FlashBack на задней панели (не требует CPU/RAM)

## 3. Автоматические оптимизации

Эти оптимизации управляются состояниями Salt и применяются автоматически при `just apply`.

### Сжатие mkinitcpio: zstd -19 на -1

**Состояние**: `states/mkinitcpio.sls`

По умолчанию CachyOS использует сжатие `zstd -19` в mkinitcpio — это даёт меньший initramfs, но медленнее распаковывается. Переход на `zstd -1` обменивает ~2-3 МБ дискового пространства на более быструю распаковку при загрузке.

- **Выигрыш**: ~1-1.5с на фазе initrd
- **Компромисс**: initramfs увеличивается с ~25 МБ до ~28 МБ (незначительно на NVMe)

### Сокетная активация libvirtd

**Состояние**: `states/desktop.sls`

libvirtd настроен на сокетную активацию вместо запуска при загрузке. Демон запускается только при подключении инструмента управления ВМ (virt-manager, virsh) к сокету.

- **Выигрыш**: 1.2с убрано с критического пути загрузки
- **Компромисс**: первая команда `virsh` после загрузки имеет задержку ~1с на запуск демона

### Отложенный запуск Loki после graphical.target

**Состояние**: `states/monitoring_loki.sls`

Loki (агрегация логов) настроен с `After=graphical.target`, поэтому запускается только после готовности рабочего стола. Это снижает конкуренцию за ввод-вывод на критическом пути загрузки, где пропускная способность NVMe наиболее важна (initrd, генераторы systemd, менеджер входа).

- **Выигрыш**: снижение конкуренции за ввод-вывод при загрузке (не измеряется напрямую по времени)
- **Компромисс**: логи из самого раннего пространства пользователя могут поступать в Loki с небольшой задержкой

### Таймаут Limine

**Состояние**: `states/kernel_params_limine.sls`

Таймаут загрузчика Limine установлен в 1с. Это уже оптимально — достаточно для перехвата нажатия клавиши для доступа к меню, но достаточно коротко, чтобы не тратить время.

## 4. Анализ параметров ядра

Система использует несколько параметров ядра для усиления безопасности. Каждый был оценён на влияние на время загрузки для определения целесообразности удаления.

### Параметры с незначительной стоимостью (оставить)

| Параметр | Стоимость при загрузке | Назначение |
|---|---|---|
| `init_on_alloc=1` | <100мс | Обнуляет освобождённые страницы памяти; предотвращает утечку данных ядра |
| `page_alloc.shuffle=1` | незначительно | Рандомизирует свободные списки аллокатора страниц; усиливает ASLR |
| `slab_nomerge` | незначительно | Запрещает объединение slab-кешей; блокирует методы эксплуатации кучи |
| `randomize_kstack_offset=on` | нулевая | Рандомизирует смещение стека ядра для каждого системного вызова |
| `vsyscall=none` | нулевая (убирает код) | Отключает устаревшую страницу vsyscall; устраняет поверхность атаки |
| `debugfs=off` | незначительно | Отключает монтирование debugfs; убирает доступ к отладочной информации ядра |

**Суммарная оценочная стоимость всех параметров безопасности: <200мс.**

Их удаление сэкономит менее 200мс при ухудшении безопасности. Не рекомендуется.

### Параметры, уже оптимизированные для скорости

| Параметр | Эффект |
|---|---|
| `nowatchdog` | Отключает таймеры watchdog — убирает периодические прерывания |
| `tsc=reliable` | Пропускает калибровку TSC по HPET/ACPI PM таймеру |
| `split_lock_detect=off` | Отключает обнаружение split-lock — убирает накладные расходы на проверку выравнивания |
| `rcupdate.rcu_expedited=1` | Использует ускоренные периоды благодати RCU — быстрее переходы состояний при загрузке |

Эти параметры уже присутствуют и вносят вклад в текущее время загрузки.

## 5. Результаты замеров

Заполняется после применения изменений и измерения через `systemd-analyze`.

| Фаза | До | После | Разница |
|---|---|---|---|
| Прошивка | 20.0с | | |
| Загрузчик | 1.3с | | |
| Ядро | 2.6с | | |
| initrd | 3.1с | | |
| Пространство пользователя | 3.4с | | |
| **Итого** | **30.5с** | | |

**Как измерить**:

```bash
# Общая разбивка времени загрузки
systemd-analyze

# Самые медленные службы
systemd-analyze blame | head -20

# Визуализация критического пути (открывает SVG)
systemd-analyze plot > /tmp/boot-plot.svg && handlr open /tmp/boot-plot.svg

# Время прошивки (если поддерживается оборудованием)
systemd-analyze --firmware-setup  # планирует перезагрузку в UEFI
```

## 6. Устранение неполадок

### Резервный initramfs

Меню загрузки Limine всегда содержит резервную запись с `initramfs-linux-cachyos-lts-fallback.img`. Этот initramfs включает все модули и не зависит от автоопределения. Если изменение конфигурации mkinitcpio ломает загрузку, выберите резервную запись в меню Limine.

### Откат через Snapper

Каждый `just apply` создаёт пару снимков btrfs. Для отката:

```bash
just rollback
```

Эта команда выполняет `snapper undochange` для последней пары снимков до/после, восстанавливая файловую систему в состояние до последнего применения Salt.

### Аварийный TTY

Если графическая сессия не запускается, переключитесь на TTY:

- **Ctrl+Alt+F2** — аварийный TTY greetd (всегда доступен)
- **Ctrl+Alt+F3-F6** — дополнительные TTY

Войдите и изучите логи:

```bash
journalctl -b --priority=err
systemctl --failed
```

### Откат изменений mkinitcpio

Если изменение сжатия mkinitcpio ломает загрузку:

1. Выберите запись **fallback** в меню загрузки Limine
2. Войдите через TTY
3. Отредактируйте `/etc/mkinitcpio.conf` — верните строку `COMPRESSION_OPTIONS`
4. Пересоберите initramfs: `sudo mkinitcpio -P`
5. Перезагрузитесь обычным образом

Альтернативно, выполните `just rollback`, если изменение было применено через Salt.
